🎶 1. Project Overview

Orpheus is a production-grade, dual-architecture audio analysis platform designed to automate the extraction of acoustic feature vectors from raw speech and audio recordings. It targets AI researchers, speech scientists, and data engineers who need structured, machine-learning-ready datasets from audio collections.

The system consists of two fully decoupled components that communicate exclusively over HTTP/JSON on localhost:

Java Desktop GUI

  • Built with Java 17 Swing + FlatLaf Dark theme
  • Drag-and-drop audio file ingestion
  • Toggle-based preprocessing and feature controls
  • Live progress bar via SwingWorker polling
  • One-click CSV and ZIP download

Python FastAPI Backend

  • FastAPI + Uvicorn REST server on port 9999
  • Librosa / SciPy / NumPy DSP engine
  • Registry-pattern plugin architecture
  • Thread-pool for non-blocking CPU work
  • Auto-creates output directories and archives

Key Capabilities

MFCC + Δ + ΔΔ (78 features) Chroma STFT (24 features) Spectral Descriptors (22 features) F0 Pitch + Jitter (5 features) RMS + Shimmer + HNR (6 features) Zero-Crossing Rate (2 features) Tempo + Duration (2 features) Butterworth High-Pass Filter Spectral Gating Noise Reduction Docker-ready · Zero-install backend

🏗️ 2. System Architecture

Orpheus follows a strict microservice-inspired separation of concerns. The Java GUI is a pure presentation layer with zero ML or DSP logic. All computation is delegated to the Python server via HTTP. This design makes each layer independently upgradeable, testable, and deployable.

🔄 3. End-to-End Data Flow

1
File Ingestion User selects .wav or .mp3 files in the Java GUI's InputSection. Paths are stored as absolute strings.
2
Configuration Assembly Java assembles an ExtractionConfig POJO (audioFiles, preprocessingSettings, featureSelection) and serialises it to JSON via GSON.
3
HTTP POST /api/config The JSON payload is sent to the FastAPI server. The endpoint validates it with Pydantic and queues the pipeline as a BackgroundTask to avoid blocking the event loop.
4
Pipeline Execution (ThreadPool) ExtractionService.run() executes in a ThreadPoolExecutor. For each audio file: resolve path → load via librosa → run preprocessors in order → save processed WAV → run all active extractors → accumulate feature row.
5
Progress Reporting A shared status dict is updated via a status_callback. The Java SwingWorker polls GET /api/status every 500 ms and renders a live progress bar for the user.
6
Export & Archive All feature rows are written to output/features/features.csv. Processed WAVs and the CSV are bundled into output/archives/speech_features.zip.
7
Download Java OutputSection triggers GET /api/download/csv or GET /api/download/zip. The server responds with a FastAPI FileResponse which the GUI saves to disk.

⚙️ 4. DSP Preprocessing Pipeline

All preprocessing steps are implemented as plugins conforming to the BasePreprocessor abstract class. Regardless of which steps the GUI enables, the execution order is always fixed and canonical to ensure deterministic, reproducible output.

Fixed Execution Order Resample → High-Pass Filter → Noise Reduction → Silence Removal → Normalisation
Step Class Algorithm / Library Key Parameters
1. Resample Resampler librosa.resample Target SR (default 22 050 Hz)
2. High-Pass Filter HighPassFilter 4th-order Butterworth (scipy.signal) Cutoff Hz (default 80 Hz)
3. Noise Reduction NoiseReducer Spectral gating — stationary mode (noisereduce) stationary=True
4. Silence Removal SilenceRemover librosa.effects.trim top_db threshold
5. Normalise Normalizer dBFS peak normalisation (NumPy) Target dBFS (default −1 dBFS)

High-Pass Filter Detail

The HighPassFilter uses a 4th-order Butterworth filter designed with scipy.signal.butter and applied via sosfilt (second-order sections form) to avoid numerical instability. The cutoff frequency is normalised to the Nyquist before design: 

# scipy.signal implementation
sos = butter(
    4,                           # filter order
    cutoff_hz / nyquist,         # normalised cutoff  (0 < Wn < 1)
    btype="high",
    output="sos",               # second-order sections
)
y_filtered = sosfilt(sos, y).astype(np.float32)

Noise Reduction Detail

Spectral gating works by estimating the noise floor from the quietest regions of the spectrum and subtracting it in the frequency domain. The stationary=True flag tells noisereduce to assume a consistent background noise profile throughout the recording: 

y_clean = nr.reduce_noise(y=y, sr=sr, stationary=True)

🧬 5. Feature Extraction Modules

All extractors inherit from BaseFeatureExtractor which mandates three responsibilities: a name property, a column_names() list, and an extract(y, sr) method. The registry pattern in ExtractionService maps JSON keys to extractor classes, enabling dynamic loading without modifying core logic.

MFCC Extractor 78 features

Computes 13 Mel-Frequency Cepstral Coefficients using librosa.feature.mfcc, then derives first-order deltas (librosa.feature.delta) and second-order delta-deltas. For each of the 39 coefficient tracks (13 × 3), mean and standard deviation are reported — yielding 78 scalar values.

Output columns: mfcc_1_mean…mfcc_13_std, mfcc_delta_1_mean…mfcc_delta_13_std, mfcc_delta2_1_mean…mfcc_delta2_13_std

Chroma Extractor 24 features

Computes an STFT magnitude spectrogram (librosa.stft), then maps it to a 12-bin pitch-class (chroma) representation via librosa.feature.chroma_stft. Bins correspond to the 12 semitones C, C#, D, D#, E, F, F#, G, G#, A, A#, B. Mean and standard deviation are reported per bin.

Output columns: chroma_1_mean…chroma_12_std

Spectral Extractor 22 features

Extracts five classical spectral shape descriptors from the Short-Time Fourier Transform. Spectral contrast is computed across 7 frequency sub-bands, each yielding a mean and std. The roll-off threshold is set at the 85th percentile of the spectral energy.

DescriptorLibrary CallFeatures
Spectral Centroidlibrosa.feature.spectral_centroidmean, std
Spectral Bandwidthlibrosa.feature.spectral_bandwidthmean, std
Spectral Roll-off (85%)librosa.feature.spectral_rolloff(roll_percent=0.85)mean, std
Spectral Contrast (7 bands)librosa.feature.spectral_contrast(n_bands=7)mean+std × 7
Spectral Flatnesslibrosa.feature.spectral_flatnessmean, std

Pitch (F0) Extractor 5 features

Uses librosa.pyin (probabilistic YIN algorithm) to estimate the fundamental frequency (F0) frame-by-frame. The search range is C2 (65.4 Hz) to C7 (2093 Hz). Frames marked as unvoiced (NaN) are excluded from statistics. Jitter is computed as the mean absolute consecutive-frame F0 perturbation, normalised by the mean F0. 

f0, voiced_flag, voiced_prob = librosa.pyin(y, sr=sr,
    fmin=librosa.note_to_hz('C2'),
    fmax=librosa.note_to_hz('C7'))

# Jitter = mean|Δf0| / mean_f0  (voiced frames only)
jitter = np.mean(np.abs(np.diff(f0_voiced))) / np.mean(f0_voiced)

Output columns: f0_mean, f0_std, f0_min, f0_max, jitter_relative

Energy Extractor 6 features

Computes RMS energy per frame via librosa.feature.rms. Shimmer (amplitude perturbation) is derived analogously to jitter — mean absolute consecutive-frame amplitude difference normalised by mean amplitude. HNR (Harmonics-to-Noise Ratio) is estimated from the peak of the normalised autocorrelation function in the lag range corresponding to 75–500 Hz, using the formula: 

HNR = 10 · log10(r_max / (1 − r_max))
# where r_max = peak normalised autocorrelation coefficient

Output columns: rms_mean, rms_std, rms_min, rms_max, shimmer_relative, hnr_db

Zero-Crossing Rate Extractor 2 features

Counts the rate at which the signal changes sign per frame using librosa.feature.zero_crossing_rate. High ZCR correlates with fricative and unvoiced consonants; low ZCR correlates with voiced speech and tonal content.

Output columns: zcr_mean, zcr_std

Temporal Extractor 2 features

Extracts duration in seconds via librosa.get_duration and estimates tempo in BPM via the onset strength envelope and librosa.feature.tempo.

Output columns: duration_sec, tempo_bpm

🖥️ 7. Java Desktop GUI

The GUI is a standard Java Swing application themed with FlatLaf 3.4 (Mac Dark variant), laid out with MigLayout 11.4.2, and uses SVG Salamander for vector icon rendering. All heavy I/O is executed off the Event Dispatch Thread via SwingWorker.

Cross-Language Communication

The ExtractionConfig Java POJO mirrors the Python Pydantic model field-for-field. GSON serialises the Java object to JSON; FastAPI + Pydantic deserialises and validates on arrival. No shared IDL or code-generation is required — the contract is maintained by keeping both models in sync manually. 

// Java side (ExtractionConfig.java)
public class ExtractionConfig {
    List<String>         audioFiles;
    Map<String, Object>  preprocessingSettings;
    Map<String, Boolean> featureSelection;
}

# Python side (main.py — Pydantic)
class ExtractionConfig(BaseModel):
    audioFiles:            List[str]
    preprocessingSettings: Dict[str, Any]
    featureSelection:      Dict[str, bool]

🌐 8. REST API Reference

The Python backend exposes five endpoints, all served by Uvicorn on port 9999. CORS is fully open (allow_origins=["*"]) to permit desktop client access.

MethodPathDescriptionResponse
GET/Health checkJSON status OK message
POST/api/configSubmit extraction job{"status":"success"}
GET/api/statusPoll pipeline progress{"progress":%, "message":"...", "is_complete":bool}
GET/api/download/csvDownload features CSVFileResponse (text/csv)
GET/api/download/zipDownload archive ZIPFileResponse (application/zip)

Concurrency Model

The pipeline is CPU-bound (librosa, scipy). To prevent blocking FastAPI's async event loop, the pipeline runs inside a ThreadPoolExecutor with 2 workers, invoked via asyncio.loop.run_in_executor. A 409 Conflict is returned if a job is already in progress.

🐳 9. Docker Deployment

Orpheus ships with a multi-stage Dockerfile and docker-compose.yml that containerise the Python backend. The Java GUI runs natively on the host and connects to the containerised server.

What Docker requires on the host machine Docker Desktop (any platform) + Java 17+ JRE. No Python, no pip, no Maven, no librosa installation needed.

Multi-Stage Build

Quick Start

# 1. Start the backend
docker-compose up --build

# 2. Copy the compiled GUI JAR (first time only)
docker cp orpheus-backend:/app/Java/SpeechDashboard.jar ./SpeechDashboard.jar

# 3. Launch the GUI on the host
java -jar SpeechDashboard.jar
